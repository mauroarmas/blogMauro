Cómo corro este blog en un contenedor de 128 MB

Un presupuesto de memoria estricto cambia cada decisión: el runtime, la base de datos y hasta cómo se hace el build. Estas son las restricciones que me impuse y lo que aprendí respetándolas.

MA Por Mauro Armas · Desarrollador y Estudiante de Ingeniería

Tenía una regla simple antes de escribir una sola línea: el sitio entero —servidor, base de datos y assets— tiene que vivir cómodo en 128 MB de RAM y 8 GB de disco. No por masoquismo, sino porque las restricciones honestas producen mejores decisiones. Cuando la memoria es cara, dejas de arrastrar dependencias “por si acaso”.

El presupuesto de memoria

Lo primero fue hacer las cuentas. Un proceso de Node.js en reposo ya pide entre 40 y 60 MB. Eso deja poco margen, así que la meta fue clara: un solo proceso, sin orquestadores, sin sidecars. El build de Next.js en modo standalone copia únicamente lo que el servidor necesita en runtime y descarta el resto.

next.config.js javascript 
// Empaqueta solo el server runtime — nada de node_modules entero
module.exports = {
  output: 'standalone',
  compress: true,
  experimental: { serverMinification: true },
}

Con eso, la imagen final pasó de cientos de megas a algo que arranca sin sudar. El truco no es exótico: es no enviar lo que no se ejecuta.

Por qué SQLite y no Postgres

Postgres es excelente, pero levantarlo significa un segundo proceso que pelea por la misma RAM. Para un blog con un CRUD de posts —lecturas frecuentes, escrituras raras— SQLite es la herramienta correcta, no el premio de consolación.

  • Vive dentro del proceso: cero memoria extra por un servidor de base de datos.
  • Con WAL activado, las lecturas no bloquean a las escrituras.
  • El backup es copiar un archivo. Cabe en un cron de tres líneas.
db.ts typescript 
import Database from 'better-sqlite3'

export const db = new Database('data/blog.db')
db.pragma('journal_mode = WAL')   // lecturas y escrituras conviven
db.pragma('synchronous = NORMAL') // durable sin pagar de más
nota Una sola conexión, reutilizada en todo el proceso. Abrir y cerrar conexiones por request es justo el tipo de derroche que 128 MB no perdona.

El resultado

El contenedor arranca en menos de un segundo y se queda holgado por debajo del límite, incluso sirviendo tráfico. No hay magia: hay restricciones tomadas en serio. Y resulta que casi todo lo que creía necesitar, no lo era.

Lo barato de operar suele ser, también, lo simple de entender.
